# tracker_for_ipad — 改善設計メモ

生徒一人ひとりが自分のiPad（横向き）で、動画から運動を解析するためのツール。
最優先は「初心者が迷わない」こと。土台は既存の `app.js` を磨く方針。
リファレンス（IPhO2023 解析アプリ / Tracker Online）は概念の参考とし、コードは現行を発展させる。

## 合意した方針

### 1. 中核インタラクション（最大の変更）
- **中央十字＋確定ボタン方式**。十字は常に画面中央固定。
- 1本指 = 映像パン、2本指 = ズーム。「確定」で現フレームに点打ち/上書き → ステップ幅ぶん自動コマ送り。
- **モード切替（微調整/トラック）は完全廃止**。過去の点の修正は「表の行タップ → そのコマへ → 十字を合わせて確定で上書き」で一貫。

### 2. 校正
- 原点・スケールも同じ「十字に合わせて確定」作法に統一。
- バグ修正: `handleCanvasClick` の裸 `calibration` → `appState.calibration`（原点/スケール設定が ReferenceError で落ちていた）。

### 3. 動画・FPS
- `requestVideoFrameCallback` で実FPSを自動測定＋手動上書き。シークも同APIで精密化し、正しい時刻 t を保証。

### 4. オートトラック
- 残すが「おまけ」に格下げ。折りたたみパネル、難しいツマミは隠す/自動化。

### 5. データ出口（アプリ内グラフ＋外部出力のハイブリッド）
- グラフ: 表・グラフ・動画の点が**連動ハイライト**。位置に加え**速度・加速度グラフ**（数値微分、FPS依存）。Tracker を参考。
- 出力: **TSVコピー** ＋ **xlsxダウンロード**（SheetJS を CDN で追加）。

### 6. レイアウト
- 横向き前提。動画を主役に大きく。画面下に「コマ戻し / 確定 / コマ送り」を常時固定（親指到達）。校正・グラフ・表は折りたたみ/引き出し。

### 7. 物体
- 2〜3物体に簡素化。数値入力でなく色付き「物体1/2/3」ボタン。

### 8. 保全・導線
- Undo（直前の点打ち/削除取消）＋ localStorage自動保存（リロード復帰、動画は再選択）。
- 手順ガイド常設（①動画 ②スケール ③原点 ④トラッキング ⑤出力）＋空スタート。
- 「サンプルで試す」ボタン: `fetch('sample.mp4')` → Blob → 読込。生徒のお試し用 兼 自動テスト用バックドア（アップロードダイアログを回避）。

### 9. 依存・配布
- フォント/アイコンは CDN 継続（Wi-Fi安定前提）。
- 開発は3ファイル（index.html / app.js / styles.css）、配布はホスティングURL（GitHub Pages 候補）。
- ローカルは `serve.py`（:8000, LAN公開, キャッシュ無効化）。
- ブラウザ実機テストは Claude の Chrome 拡張で実施。

## ビジュアル・アイデンティティ（"計器 / ストロボ"）
主題（動画運動解析）の世界＝ストロボ運動写真＋計測器。ランディングではなく"計器"として作る。

- **配色**: `--ink #0F1216`（暗室グラファイト）/ `--panel #171B21` / `--line #2B333D`（ヘアライン＝方眼）/
  `--text #E6EAEF` / `--muted #8A95A3` / **`--trace #FFB627`（シグナル・アンバー）** / `--cyan #5AA9E6`。
- **2色のロジック**: アンバー＝**操作とライブ計測**（確定ボタン・照準レティクル・現ステップ・FRAME/FPS読み出し・
  スライダ・再生グリフ）。シアン＝**計器の枠と校正**（ビューファインダのコーナーブラケット・節タイトルの目印・
  原点/スケール値）。大胆さは「アンバーの確定ボタン」一点に集中、周囲は静かに。
- **書体**: IBM Plex Sans JP（工学のための書体・日本語＋ラテン）＋ IBM Plex Mono（計器の数値読み出し）。
  データ（t/x/y・コマ・FPS）は全てモノスペースでメーター化。
- **Signature**: ビューファインダ。動画をL字コーナーブラケット＋薄い方眼の"計器スクリーン"に収め、
  照準レティクルをアンバーで発光。位置マーカーはストロボの残像のように尾を引く（減衰ドット）。
- **品質の床**: レスポンシブ（〜768pxまで）、`:focus-visible` 可視、`prefers-reduced-motion` 尊重、`color-scheme: dark`。

## 実装順（段階的・各段階で動作確認＆commit）
1. **Stage1**: バグ修正 ＋ 中央十字方式 ＋ 下部固定バー（中核が動く最小形）
2. **Stage2**: FPS精密化（requestVideoFrameCallback）
3. **Stage3**: グラフ強化 ＋ データ出力（TSV/xlsx）
4. **Stage4**: 保全 / 導線 / 物体 / オートトラック格下げ

# 改修計画 2026-07 — iPad実機の三大障害の解消＋ストロボ写真

2026-07-02 のグリルセッションで合意。サポート下限は **iPadOS 15.4**
（requestVideoFrameCallback と svh/dvh の両対応ライン）。

## 診断（確定済み）

| 症状 | 原因 |
|---|---|
| 読み込みが異様に遅い | `startFrameScan()` が読込直後に全フレームをシーク走査（1シーク100〜500ms × 数百回＝数分） |
| 進む/戻るでコマがずれる | `seekToFrame()` が投げっぱなしシークで完了を待たず状態更新。Safariは連続シークを間引く。再生中は `floor(t×fps)` 逆算で時刻表と別座標系 |
| レイアウト崩れ | `100vh`（iOS Safariのツールバー裏問題）＋ `max-width:1024px` breakpoint がiPad横向き(=1024px)を巻き込み縮小レイアウトに落とす |

## 決定事項

- **対象動画**: iPad通常撮影 / スロモ(VFR) / 教員配布 / 長尺の一部解析 — 全部対応。
- **時刻表**: 全フレーム走査を廃止。**mp4box.js（自前ホスティング）でコンテナから
  実サンプル時刻(cts)を瞬時に取得**。VFR・スロモもそのまま正確。パース不能な形式のみ
  現行のシーク走査へフォールバック（rVFC無し端末の劣化系も維持）。
- **シーク**: Promiseベースの**直列化キュー**に一本化。rVFCの mediaTime で「実際に
  表示されたフレーム」を確認してから `currentFrame` を確定。再生中のコマ番号も
  時刻表の二分探索に統一。
- **解析範囲指定**: シークバーにイン/アウト点ハンドル。範囲外はコマ送り・表・グラフから除外。
- **重複フレーム除外**: 読込時ではなく**解析範囲の確定後にその範囲だけ**ピクセル比較で
  実行（数秒で完了、50cm精度の根拠を維持）。〔仮採用・要ユーザー確認〕
- **レイアウト**: `100svh`(+vhフォールバック) / `env(safe-area-inset-*)` を `max()` で /
  breakpointはデバイス決め打ちをやめコンテンツ基準で引き直し（Split View対応）/
  canvasに `touch-action:none`＋自前ジェスチャ、他UIは `manipulation` /
  input は16px以上・`user-scalable=no` 廃止 / canvas解像度は4096²でクランプ。
- **CDN脱却**: フォント(woff2)・アイコン・SheetJS・mp4box.js をリポジトリ同梱
  （学校MDMフィルタ対策・教室Wi-Fi安定化）。
- **ストロボ写真**: **追跡点周辺の円形パッチ切り貼り方式**（背景の明暗に依存しない）。
  基準フレームの上に各確定点周辺を重ねて「残像列」を合成。
  出力は PNGダウンロード＋画面内プレビュー（コマ間引き・パッチ半径スライダ）。
  フレームは1枚ずつシークして合成しキャッシュしない（canvasメモリ384MB上限対策）。
  〔プレビューUIの詳細は仮採用・要ユーザー確認〕
- **localStorage自動復帰の廃止→動画フィンガープリント一致時のみ復元提案**（2026-07-02追加合意）

## 実装順（各段階でiPad実機確認＆commit）

1. **StageA レイアウト土台**: svh / safe-area / breakpoint再設計 / touch-action /
   16px input / DPRクランプ。既存機能に手を入れず見た目の崩れだけ直す。 — **済 2026-07-02**
2. **StageB 動画エンジン**: シーク直列化キュー＋rVFC確認（ずれの根治）→
   mp4box.js時刻表（遅さの根治）→ 走査はフォールバックへ格下げ。 — **済 2026-07-02**
3. **StageC 範囲指定＋CDN脱却**: イン/アウト点、範囲確定時の重複除外、アセット同梱。 — **済 2026-07-02**
4. **StageD ストロボ写真**: パッチ合成エンジン＋プレビューダイアログ＋PNG出力。 — **済 2026-07-02**

### 実装時の計画からの逸脱（意図的）
- **DPRクランプは見送り**: 現行canvasはDPRを掛けない1x実装のため上限超過のリスクが
  存在しない。高精細化(DPR対応)自体を将来やる時にクランプも一緒に入れる。
- **本文フォント(IBM Plex)はCDN継続**: JPフォントのサブセット群は数MB×多数ファイルで
  同梱コストが大きい。ブロック時はシステムフォントに劣化(機能は無傷)。
  アイコン/SheetJS/mp4boxは同梱済みで、MDMフィルタ下でも機能は全て動く。
- **未実施**: iPad実機での確認（svh/safe-area/タッチ挙動・コンテナ解析の速度・
  ストロボの見栄え）。テストは実Chrome CDPのみ。

## 調査メモ（出典はグリルセッションのサブエージェント調査）

- `currentTime` 単体のフレーム精密シークは構造的に不可能（WebKit既知バグ＋仕様非保証）。
  シーク→rVFC確認→必要なら補正、が定番。
- WebCodecs(iOS16.4+)は本アプリには過剰装備。HEVCのVideoDecoderは不安定報告あり。
- iPad撮影動画は HEVC/.mov がデフォルト。mp4box.js はISOBMFF系(.mov含む)を概ね
  パース可能だが、失敗時フォールバック必須。
- iOS Safari の canvas 総メモリ上限 ≈ 384MB（超過は「透明描画」の静かな失敗）。
- PWA化は今回見送り（7日間ストレージ消去・Safari版とのデータ分離の運用リスク）。

# 改修計画 StageE (2026-07-13) — スロー動画対応・キャッシュ可視化・スムージング・ストロボ等間隔化

実データ(IMG_0128.MOV、iPad Air 11-inch M4で撮影)を分析したところ、測定した重力加速度が
実測よりずっと小さい(数十分の一)という異常が見つかり、原因調査の結果ここに至った。

## 診断（確定済み）

| 症状 | 原因 |
|---|---|
| 加速度・重力加速度が実測よりずっと小さい | iPad/iPhoneの「スロー」撮影を書き出したファイルは、コンテナが見かけ上のfps(例:30fps)を名乗る一方、実際の撮影fpsはもっと高い(例:240fps)。この「フラット化」は不可逆で、真のfpsはファイルのメタデータからは復元できない（`ffprobe`でも container fps=30 としか出ない） |
| 速度・加速度グラフがガタつく | +1コマ・+10コマ等ステップ幅を混在させると、前後区間の時間比が極端(10倍超)になる点で、3点厳密内挿の数値微分がノイズを増幅する |
| 「今使っているアプリが正しいバージョンか」が分からない | app.jsにバージョン表示・キャッシュ回避の仕組みが皆無だった。実際、今回の書き出しデータは2026-07-03の不等間隔対応修正(`8cd5470`)より前のロジックで計算されていたことが事後判明した |
| ストロボ写真が等間隔に見えない | `strobePoints`が打刻順インデックスでN個おきに間引いていたため、ステップ幅混在時は実際には不等間隔になる |

## 決定事項

- **物理時間とシーク時間を分離**: `videoFps`(シーク用・実コンテナfps)はそのまま維持し、
  新設の`physicsFpsMultiplier`(既定1)を`frameTimeOf`(物理時間)にだけ掛ける。
  `seekTimeOf`(動画シーク・コマ送り)は一切変更しない。混同すると動画シークそのものが壊れる
- **スロー倍率は自動ヒント＋手動入力の併用**: コンテナに`full-frame-rate-playback-intent`等
  スロー痕跡があれば、厳密なボックスパースはせず軽量なバイト列検索でヒント表示するに留める。
  実際の撮影fpsは常に手動入力できる(120fps/240fpsプリセット+任意入力+「わからない/通常」)。
  真の撮影fpsはファイルから原理的に復元不可能な場合があるため、自動検出だけに頼らない
- **速度・加速度は窓付き回帰スムージングを既定ON**: 各点の前後2点=計5点で2次関数を
  最小二乗フィットし、その微分値を採用。窓がちょうど3点(データ点が少ない/端点)なら
  旧来の3点厳密内挿(`derivExact`)と完全一致(後方互換)。「生データ表示」トグルで
  いつでも厳密差分に切替可能（精度検証・上級者向け）。グラフ・エクスポート双方に反映し、
  エクスポート(xlsx/TSV)にはどちらのモードで出力したかをメモ行として明記する
- **ストロボ写真は時間的な等間隔選択に変更**: 打刻順インデックスのN個おきではなく、
  物理時間でN/実効fps刻みの目標時刻に最も近い点を選ぶ。近くに点が無い回は静かにスキップし、
  遠い点を無理に採用しない（「見た目は等間隔だが実は違う」を避けるため）
- **バージョン表示**: `app.js`先頭に手動更新の`APP_VERSION`定数を追加し、デバッグ
  コンソールに常時表示。`index.html`側は`app.js?v=...`でキャッシュ回避を試みる
  （完全な保証はできないので、あくまで「今どの版で動いているか」を確認できることが主目的）

## 実装順

1. **StageE-1 バージョン表示**: `APP_VERSION`定数＋画面表示＋キャッシュバスティングクエリ。 — **済 2026-07-13**
2. **StageE-2 物理時間分離＋スロー倍率補正**: `physicsFpsMultiplier`／スロー痕跡ヒント検出／
   「スロー設定」ダイアログ（プリセット+手動入力）。 — **済 2026-07-13**
3. **StageE-3 ストロボ等間隔化**: `strobePoints`を時間ベースの選択に変更。 — **済 2026-07-13**
4. **StageE-4 窓付き回帰スムージング**: `computeKinematics`に最小二乗フィットを追加、

### 実装時の計画からの逸脱（意図的）
- **スロー痕跡の自動検出はボックスツリーの厳密パースをしない**: mp4box.jsの現行利用は
  サンプル時刻の取得のみで、QuickTimeのカスタムメタデータ(keys/ilst)は読んでいない。
  正確に値まで読むには`keys`と`ilst`の対応関係を追う本格的なパーサーが要るため、
  「キー文字列がファイル中に存在するか」だけを見る軽量な力技(バイト列検索)に留めた。
  ヒントが外れても実害はヒント表示の有無だけ（手動入力は常に可能）
- **未実施**: iPad実機での確認（スロー動画の実際のヒント検出率・ダイアログの使い勝手）。
  テストはnode(logic)と実Chrome CDP(E2E/精度)のみ。
